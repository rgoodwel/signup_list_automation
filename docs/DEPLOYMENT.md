## 1. Architecture Overview

### App Type
**Static Single-Page Application (React + Vite)** – All data is stored in browser `localStorage`. No server, no database, no API calls to AWS. This dramatically simplifies and cheapens the architecture.

### Target Architecture

```
  Browser
     │  HTTPS
     ▼
┌─────────────────────┐
│   Amazon CloudFront │  CDN, TLS termination, security headers,
│   (Global PoPs)     │  SPA 404→index.html routing, cache
└─────────┬───────────┘
          │  OAC (Origin Access Control, sigv4)
          ▼
┌─────────────────────┐
│   Amazon S3 Bucket  │  Private bucket, stores Vite dist/ assets
│   (us-east-1)       │  Versioning enabled, AES-256 encryption
└─────────────────────┘

Optional (custom domain path, deferred for now):
  Route 53 → CloudFront alias
  ACM (us-east-1) → CloudFront viewer cert
```

### Why These Services?

| Service | Used? | Reason |
|---------|-------|--------|
| **S3** | ✅ Yes | Static asset hosting (private origin) |
| **CloudFront** | ✅ Yes | CDN, HTTPS, security headers, SPA routing |
| **ACM** | ✅ Optional | TLS cert for custom domain |
| **Route 53** | ✅ Optional | DNS for custom domain |
| **IAM** | ✅ Yes | Least-privilege for GitHub Actions OIDC |
| **DynamoDB** | ✅ State only | Terraform state locking table (not app data) |
| **ALB** | ❌ No | No server-side compute needed |
| **Lambda/EC2/ECS** | ❌ No | Purely static SPA |
| **Cognito/Auth0** | ❌ Not yet | Admin is client-side only (see security note) |

### Security Note on Admin Login
The current app uses a client-side admin login (name matching only). This is appropriate for a small private league. If you need stronger security in the future, the recommended path is:
1. Add **Amazon Cognito** user pool with an admin group
2. Move admin authorization checks server-side (Lambda@Edge or a small API)
3. This would require adding ALB + Lambda to the architecture

## 2. Terraform Project Structure

```
infra/
├── bootstrap/                    # Run ONCE per account – state backend + OIDC
│   ├── main.tf                   # S3 bucket, DynamoDB table, OIDC provider, IAM role
│   ├── variables.tf
│   ├── outputs.tf
│   └── bootstrap.tfvars.example
│
├── modules/
│   └── static-site/              # Reusable S3 + CloudFront module
│       ├── main.tf
│       ├── variables.tf
│       ├── outputs.tf
│       └── versions.tf
│
└── envs/
    ├── dev/                      # Dev environment
    │   ├── main.tf               # Calls static-site module
    │   ├── variables.tf
    │   ├── outputs.tf
    │   ├── terraform.tfvars.example
    │   └── backend.hcl.example
    └── prod/                     # Prod environment (same structure)
```

### Key Design Decisions
- **Per-env state keys**: `dev/terraform.tfstate`, `prod/terraform.tfstate` — all in one state bucket but isolated.
- **OAC over OAI**: Origin Access Control is the modern AWS recommendation (OAI is legacy).
- **`PriceClass_100`**: US/EU CloudFront PoPs only — cheapest option, fine for most golf leagues.
- **Security headers**: Applied via CloudFront response headers policy (CSP, HSTS, X-Frame-Options, etc.)

## 3. IAM & Security

### GitHub Actions OIDC Role (created by bootstrap)

**Trust Policy** (auto-generated by bootstrap):
```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "Federated": "arn:aws:iam::ACCOUNT_ID:oidc-provider/token.actions.githubusercontent.com"
    },
    "Action": "sts:AssumeRoleWithWebIdentity",
    "Condition": {
      "StringEquals": {
        "token.actions.githubusercontent.com:aud": "sts.amazonaws.com"
      },
      "StringLike": {
        "token.actions.githubusercontent.com:sub": "repo:rgoodwel/signup_list_automation:*"
      }
    }
  }]
}
```

**Permission Scope**: See `infra/bootstrap/main.tf` for the full policy. Key permissions:
- S3: state bucket + app bucket management
- CloudFront: distribution + invalidation
- ACM: certificate lifecycle
- Route 53: DNS record management
- IAM: manage roles created by Terraform
- DynamoDB: state lock table only

### S3 Bucket Security
- Public access completely blocked
- AES-256 server-side encryption
- Versioning enabled (supports rollback)
- Access only via CloudFront OAC (bucket policy enforced)

### CloudFront Security
- TLS 1.2+ enforced (TLSv1.2_2021 security policy)
- HTTP → HTTPS redirect
- Security response headers (HSTS, CSP, X-Frame-Options, X-XSS-Protection)
- IPv6 enabled

## 4. GitHub Actions CI/CD

### Workflow Summary

| Workflow | Trigger | Purpose |
|----------|---------|---------|
| `ci.yml` | Every push/PR | Build app, terraform fmt/validate, tflint, checkov |
| `terraform-plan.yml` | PRs to main (infra changes) | Run `terraform plan`, post to PR |
| `terraform-apply.yml` | Push to main (infra changes) or manual | `terraform apply` per env |
| `deploy.yml` | Push to main (app changes) or manual | Build + S3 sync + CF invalidation |

### Required GitHub Secrets

Go to **Settings → Secrets and variables → Actions** and add:

| Secret | Value | Scope |
|--------|-------|-------|
| `AWS_ROLE_ARN` | Output from bootstrap (role ARN) | Repository |
| `AWS_REGION` | `us-east-1` | Repository |
| `TF_STATE_BUCKET` | State bucket name | Repository |
| `TF_STATE_LOCK_TABLE` | `golf-league-tf-lock` | Repository |
| `TF_VAR_BUCKET_NAME_DEV` | Dev S3 app bucket name | Repository |
| `TF_VAR_BUCKET_NAME_PROD` | Prod S3 app bucket name | Repository |
| `S3_BUCKET_DEV` | Dev S3 app bucket name | `dev` environment |
| `S3_BUCKET_PROD` | Prod S3 app bucket name | `prod` environment |
| `CF_DISTRIBUTION_ID_DEV` | Dev CloudFront dist ID | `dev` environment |
| `CF_DISTRIBUTION_ID_PROD` | Prod CloudFront dist ID | `prod` environment |

### Required GitHub Environments

Go to **Settings → Environments** and create:

| Environment | Protection | Notes |
|-------------|------------|-------|
| `dev` | None | Auto-deploys from main |
| `dev-plan` | None | For plan jobs |
| `prod` | **Required reviewer(s)** | Manual dispatch only |
| `prod-plan` | Required reviewer(s) | For plan jobs |

## 5. First-Time Setup Runbook

### Prerequisites
- AWS account with admin access (for bootstrap only)
- AWS CLI v2 installed and configured (`aws configure`)
- Terraform ≥ 1.7 installed
- GitHub repo access

---

### Step 1: Bootstrap AWS Resources (run once)

```bash
# 1a. Clone and enter bootstrap dir
cd infra/bootstrap

# 1b. Copy and fill in your values
cp bootstrap.tfvars.example bootstrap.tfvars
# Edit bootstrap.tfvars:
#   state_bucket_name = "golf-league-tf-state-<your-initials-or-random>"
#   github_repo       = "rgoodwel/signup_list_automation"

# 1c. Initialize and apply
terraform init
terraform apply -var-file=bootstrap.tfvars

# 1d. Save outputs – you'll need these for GitHub secrets
terraform output
```

> ⚠️ The bootstrap uses **local state**. Do NOT delete your bootstrap directory or `.tfstate` file. The bucket and role it creates are used by all subsequent Terraform runs.

---

### Step 2: GitHub Repository Setup

```bash
# 2a. In GitHub: Settings → Environments
#     Create: dev, dev-plan, prod, prod-plan
#     Add required reviewers to "prod" and "prod-plan"

# 2b. Settings → Secrets and variables → Actions
#     Add all secrets from the table above.
#     Use the terraform outputs from Step 1 for:
#       AWS_ROLE_ARN    → github_actions_role_arn output
#       TF_STATE_BUCKET → state_bucket_name output
```

---

### Step 3: Choose S3 Bucket Names

Choose unique names now (they must be globally unique in S3):
```
golf-league-dev-site-<4-random-chars>
golf-league-prod-site-<4-random-chars>
```
Add these as GitHub secrets (`TF_VAR_BUCKET_NAME_DEV`, etc.) and also as `S3_BUCKET_DEV`, etc.

---

### Step 4: Deploy Dev Infrastructure

```bash
# Option A: Push a change to main that touches infra/
git commit --allow-empty -m "chore: trigger dev terraform apply"
git push origin main

# Option B: Manual dispatch
# GitHub → Actions → "Terraform Apply" → Run workflow → environment: dev → confirm: apply
```

After apply completes:
```bash
# Get the CloudFront distribution ID from Terraform outputs
# (visible in the GitHub Actions job summary or via AWS Console)
# Add it as CF_DISTRIBUTION_ID_DEV secret
```

---

### Step 5: Deploy the App to Dev

```bash
# Option A: Push a change to src/ or trigger via Actions
# GitHub → Actions → "Deploy" → Run workflow → environment: dev

# Option B: Automatic – any push to main touching src/ triggers deploy
```

#### Verify Dev Deployment
1. Open AWS Console → CloudFront → find your distribution
2. Visit the CloudFront URL: `https://<id>.cloudfront.net`
3. App should load, signup form should work, data saves to localStorage
4. Check browser DevTools → Network → confirm assets served from CloudFront

---

### Step 6: Promote to Prod

```bash
# Requires manual approval from environment reviewers

# Terraform (first time or infra changes):
# GitHub → Actions → "Terraform Apply" → environment: prod → confirm: apply

# App:
# GitHub → Actions → "Deploy" → environment: prod
# → Reviewer receives approval request → approves → deploy runs
```

---

### Step 7: Custom Domain (Optional)

1. Register/transfer your domain to Route 53 (or set up a hosted zone for an external domain)
2. Get the hosted zone ID from Route 53 console
3. Update GitHub secrets and `terraform.tfvars` with:
   ```
   custom_domain   = "golf.yourdomain.com"
   route53_zone_id = "ZXXXXXXXXXXXXX"
   ```
4. Re-run terraform apply → ACM cert created, DNS validation records added automatically
5. Wait for cert to validate (usually 2-5 minutes)
6. CloudFront alias and Route 53 A record are created automatically

## 7. Cost Estimate

For a small private golf league (low traffic):

| Service | Config | Est. Monthly Cost |
|---------|--------|-------------------|
| S3 (2 envs) | ~50 MB assets, <1 GB transfer | ~$0.01–$0.05 |
| CloudFront | PriceClass_100, <1 GB/month transfer | ~$0.08–$0.50 |
| DynamoDB | On-demand, state lock only (minimal ops) | ~$0.00–$0.01 |
| ACM | Free (always free) | $0.00 |
| Route 53 | $0.50/hosted zone + $0.40/million queries | ~$0.50–$1.00 |
| **Total** | | **~$1–$2/month** |

> CloudFront has a **1 TB free tier** for the first 12 months. For a small league, you may pay nothing for the first year.

## 8. Rollback & Disaster Recovery

### App Rollback (fast)
```bash
# Re-deploy a previous Git commit's build:
# GitHub → Actions → "Deploy" → select the previous commit's workflow run
# Or re-trigger with the SHA of a known-good commit

# S3 versioning is enabled – you can restore previous files via console or CLI:
aws s3api list-object-versions --bucket golf-league-prod-site-xxxx --prefix index.html
aws s3api get-object --bucket golf-league-prod-site-xxxx \
  --key index.html --version-id <VERSION_ID> index.html.backup
```

### Infrastructure Rollback
```bash
# Terraform state is versioned in S3. To revert to a previous infra state:
# 1. Find the previous state version in S3 console (versioning enabled)
# 2. Restore it as the current version
# 3. Run terraform apply to reconcile
```

### CloudFront Cache Issues
```bash
# Force invalidation:
aws cloudfront create-invalidation \
  --distribution-id <CF_DIST_ID> \
  --paths "/*"
```

### State Lock Stuck
```bash
# If a Terraform run was interrupted and the lock is stuck:
terraform force-unlock <LOCK_ID>
# Lock ID is shown in the error message
```

### Data Loss Note
> This app stores all data in **browser localStorage**. There is no server-side backup. If a user clears their browser data, their signup data is lost. Consider adding a backend (Lambda + DynamoDB) if data persistence is critical for future seasons.
